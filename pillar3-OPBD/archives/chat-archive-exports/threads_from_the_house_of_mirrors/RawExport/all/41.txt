You’re not imagining it — WordPress (especially the WordPress.com editor with the block-based 2025 theme) can start behaving oddly after a long editing session.  

Here’s what’s going on under the hood and what usually helps:

---

### 🧠 Why It Happens
- **Memory creep in the browser:** The Gutenberg block editor builds a live React tree. After a while, all those state changes pile up in memory, and the page gets sluggish or unresponsive.  
- **Autosave loops:** Every few minutes WordPress autosaves. If your connection hiccups or the page is left open for hours, queued autosaves can collide or hang.  
- **Preview and block rendering:** Each time you click in/out of blocks, it re-renders previews and loads assets dynamically. That cumulative rendering overhead grows with long sessions.  
- **Browser cache or plugin conflict:** If you’re logged in for a while, cached JavaScript and stale cookies can start misfiring, especially with third-party blocks or embeds.

---

### 🧩 What Usually Fixes It
1. **Quick reset:**  
   Save draft → close the tab → reopen the editor fresh. That clears memory and usually restores smooth behavior.  
2. **Hard reload:**  
   `Ctrl + Shift + R` (or `Cmd + Shift + R`) forces WordPress to reload its JS and CSS bundles from scratch.  
3. **Split long edits:**  
   If you’re making major structural changes, use *List View* or edit in small passes instead of one long marathon session.  
4. **Check browser load:**  
   If you’re in Chrome or Edge, open Task Manager (`Shift + Esc`) and check if “WordPress Editor” is eating RAM; if so, restarting helps.  
5. **Disable unneeded plugins while editing** (if you’re on a plan that allows them). Even Jetpack’s block library can cause lag over time.  
6. **Periodic logout/login:**  
   Session tokens and cached REST API auth sometimes degrade after hours of use.
